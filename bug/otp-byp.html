Date: 28/08/21

Let's consider an application as example.com. The application features a signup/register page where users are prompted to input their mobile number. Upon providing the mobile number, the server sends an OTP code to that number.



Subsequently, the user encounters an OTP submission page, where they can input the received OTP and proceed with the registration process.

I copied the response and stored it for further analysis. Subsequently, I successfully created an account using a valid OTP


Now, the actual attack begins. I utilized the phone number 734257XXXX, for which I have no access. Before proceeding, I made some modifications to the response obtained from the previous phone number. Specifically, I replaced the phone number with our victim's phone number.


I opened the registration page, entered the victim's phone number, and was redirected to the OTP submission page. I inputted a random OTP and pressed enter.

The response indicated a login error, suggesting that the random OTP input did not match the expected validation criteria.

At this point, I decided to replace the error response with the previously edited response, and I forwarded the request. This action aims to manipulate the system by substituting the error response with the modified response obtained earlier.
